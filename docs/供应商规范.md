# 供应商规范

本文档定义供应商层的职责、变更边界，以及与渠道层、分组层的关系。

## 文档信息

- 最后核对日期：2026-05-16
- 关键代码路径：
  - `internal/admin/model/provider_model.go`
  - `internal/admin/model/provider_migration_snapshot.json`
  - `internal/admin/model/provider_migration_data.go`
  - `internal/admin/model/provider_model_resolver.go`
  - `internal/admin/model/migration_runner.go`

## 摘要

本层拥有：

1. 供应商本身。
2. 供应商模型。
3. 模型官方价格、官方端点范围、能力标签、状态。

本层不拥有：

1. 渠道模型是否启用。
2. 渠道端点是否启用。
3. 分组是否暴露某模型。

本层修改入口：

1. 供应商官方网站或官方文档确认后的 migration。
2. 管理员对供应商与供应商模型的显式维护。

## 1. 本层职责

供应商层只负责供应商和供应商模型相关的官方信息，包括：

1. 供应商本身。
2. 供应商模型。
3. 模型官方价格。
4. 模型官方端点范围。
5. 模型能力标签、状态、归属等字段。

供应商层不是渠道端点启用真值层。

供应商层信息来源必须明确：

1. 供应商模型名称、价格、端点、状态等信息，必须来自供应商官方网站或官方文档。
2. 官方没有明确给出的信息，不能写成“官方已确认”，必须保留未确认状态，或明确标注来源和口径。
3. 第三方整理站、社区帖子、非官方镜像页面，不能作为最终依据。

## 2. 本层不拥有的配置

供应商层明确不拥有：

1. 某个渠道是否启用某模型。
2. 某个渠道是否启用某个端点。
3. 某个分组是否对外暴露某模型。
4. 某个分组是否可路由到某个渠道。

这些都属于下层配置。

## 3. 当前实现现状

当前供应商模型数据主要落在 `provider_models`，其中：

1. 端点范围定义该供应商模型官方支持哪些端点。
2. 官方价格、模型能力标签等信息用于渠道建模、定价展示、能力补全。

当前代码里，渠道层会读取供应商模型的官方端点范围，再生成渠道端点可维护列表。

因此供应商层提供的是官方信息和可选范围，不是渠道或分组的人工配置真值。

## 4. 标准定义

### 4.1 官方信息与真值边界

1. 供应商模型上的端点范围只是官方可选范围，不是渠道端点启用真值。
2. 供应商层的官方价格不等于渠道最终生效价格。
3. 供应商层的模型存在，不等于所有渠道都接入了该模型。
4. 对文本生成类模型，供应商层只能声明 `/v1/chat/completions`、`/v1/responses`、`/v1/messages` 中官方支持的一个或多个端点。
5. Router 最终是否对外暴露这些文本端点，还必须继续和渠道端点启用状态、分组可达渠道做交集判断。
6. 供应商模型 ID 不使用 `latest` 这类浮动别名；优先使用官方当前推荐的稳定模型 ID。若官方定价只绑定到日期快照，则使用对应日期快照 ID。

### 4.2 模型 tags 规范

供应商模型能力统一使用 `tags` 表达。供应商模型 API 的请求和响应只使用 `tags` 表达模型能力。

`tags` 是字符串数组，用于描述模型能力集合。一个模型可以同时拥有多个标签，例如支持视觉输入的文本模型可以是 `["text", "vision"]`，支持函数调用和结构化输出的文本模型可以是 `["text", "tool_calling", "structured_output"]`。

`tags` 分为两类：

1. 基础形态标签：决定模型的主要路由、端点候选、默认计价单位等基础行为。
2. 能力补充标签：描述模型在基础形态之上的细粒度能力，不能单独决定模型形态。

供应商模型必须显式包含至少一个基础形态标签。缺少基础形态标签时，新增或编辑供应商模型应直接报错，不允许由系统自动猜测或补默认值。

| Tag | 分类 | 含义 | 典型端点 | 计费/路由含义 | 备注 |
| --- | --- | --- | --- | --- | --- |
| `text` | 基础形态 | 文本生成、对话、推理等文本类模型 | `/v1/chat/completions`、`/v1/responses`、`/v1/messages` | 默认按 token 类价格计费；文本端点暴露仍需和渠道真实启用端点取交集 | 支持视觉输入的多模态对话模型通常仍应带 `text`，再追加 `vision` |
| `image` | 基础形态 | 图片生成、图片编辑，或以图片任务为主的模型 | `/v1/images/generations`、`/v1/images/edits`，必要时可包含 `/v1/responses` | 默认按图片或供应商官方组件计费；端点必须来自供应商官方支持范围 | 视觉理解模型如果主要通过文本对话端点调用，不应仅因为名字含 `vl` 就写成 `image`，应按官方调用方式判断 |
| `audio` | 基础形态 | 语音合成、语音识别、音频理解、实时音频等模型 | `/v1/audio/speech`、`/v1/realtime`，必要时可包含文本端点 | 默认按字符、分钟、token 或组件价格计费，以官方价格口径为准 | `realtime` 只是补充能力，音频实时模型通常是 `["audio", "realtime"]` |
| `video` | 基础形态 | 视频生成或视频处理模型 | `/v1/videos` | 默认按秒、任务或官方组件价格计费 | 只表达视频任务形态，不表达是否支持图文混合输入 |
| `embedding` | 基础形态 | 向量化、检索嵌入模型 | `/v1/embeddings` | 默认按 token 类价格计费，具体按官方价格口径 | 多模态 embedding 仍以 `embedding` 为基础形态，可追加 `vision`、`audio` 等标签时需谨慎，避免混淆路由形态 |
| `tool_calling` | 能力补充 | 支持工具调用、函数调用或等价能力 | 通常依附文本端点 | 不单独决定路由和计价；用于客户端能力展示和后续策略判断 | 必须和基础形态标签一起出现 |
| `reasoning` | 能力补充 | 具备推理、思考链、深度推理等官方声明能力 | 通常依附文本端点 | 不单独决定路由和计价；可用于展示、筛选和策略 | 不代表一定暴露推理 token 细节，计费仍看价格组件 |
| `vision` | 能力补充 | 支持图片输入、视觉理解或图文多模态理解 | 通常依附 `/v1/chat/completions`、`/v1/responses`、`/v1/messages` | 不单独决定路由和计价；表示输入能力 | 图片生成模型应使用 `image`；视觉理解文本模型应使用 `text + vision` |
| `realtime` | 能力补充 | 支持实时会话、低延迟双向交互或实时音频协议 | `/v1/realtime` | 不单独决定模型形态；实时端点仍需供应商和渠道均支持 | 常见组合是 `audio + realtime` 或 `text + realtime` |
| `structured_output` | 能力补充 | 支持 JSON schema、结构化输出或等价受约束输出能力 | 通常依附文本端点 | 不单独决定路由和计价；用于客户端能力展示和请求策略 | 是否支持严格 schema 仍以端点策略和供应商文档为准 |

规则：

1. `tags` 必须来自官方文档、官方控制台、供应商 API 明确返回，或管理员明确维护。
2. `tags` 不用于替代 `supported_endpoints`；标签描述能力，`supported_endpoints` 描述官方端点范围。
3. `tags` 不用于替代价格组件；价格和倍率仍以供应商价格、渠道覆盖价格和分组倍率为准。
4. `/v1/models.tags` 面向第三方应用集成，表达模型能力集合；`/v1/models.supported_endpoints` 仍表达当前分组真实可调用端点集合。
5. 新增标签前必须先补本文档表格，再补后端规范化逻辑、管理端选项和测试。

### 4.3 migration 规范

供应商层 migration 的职责是维护供应商和供应商模型的官方信息。

允许的行为：

1. 新增供应商。
2. 新增供应商模型。
3. 根据供应商官方网站或官方文档，修正模型官方价格、能力标签、状态、端点范围等信息。
4. 为既有供应商和供应商模型官方信息做幂等修复。

禁止的行为：

1. 直接为渠道自动启用模型。
2. 直接为渠道自动启用端点。
3. 直接删除渠道层已有 `channel_models` 或 `channel_model_endpoints`。
4. 直接删除分组层已有 `group_models` 或 `group_model_channels`。

### 4.4 新增模型规范

当供应商新增模型时：

1. migration 可以根据供应商官方网站或官方文档补齐供应商模型与相关官方信息。
2. 新模型可以进入后续渠道 refresh 的上游模型比对范围。
3. 但该模型是否真正被某渠道启用，必须由渠道层确认。

### 4.5 模型下线规范

当供应商模型下线时：

1. migration 只能根据供应商官方网站或官方文档，更新该模型的状态字段。
2. 当前状态字段只允许两个值：
   - `active`：模型正常可用
   - `deprecated`：模型已官方下线、停止推荐或只保留兼容
3. 供应商模型下线时，migration 必须把状态更新为 `deprecated`，不能直接删除。
4. migration 可以让后续 refresh 不再把它视为新的上游模型比对范围。
5. 删除必须由管理员手动执行。
6. 后端执行删除前，必须检查该模型是否仍在渠道中启用；如果仍有渠道启用该模型，则不允许删除。
7. 渠道和分组是否下线该模型，由管理员显式处理。

## 5. 与渠道层的关系

供应商层对渠道层提供的只有两类信息：

1. 供应商模型的官方信息。
2. 端点范围、官方价格、模型能力标签、模型状态等信息。

渠道层消费这些数据时，必须遵守：

1. 可以把这些官方信息渲染为“可选择的模型与端点”。
2. 不能把这些官方信息直接当成渠道已启用真值。
3. 对供应商模型官方信息的修正不能覆盖渠道层人工开关。

## 6. 与分组层的关系

供应商层不直接驱动分组暴露。

也就是说：

1. 供应商模型里有某模型，不代表分组一定暴露它。
2. 供应商模型里移除某模型，也不代表分组配置会被自动删除。
3. 分组是否仍暴露该模型，取决于分组配置与渠道层实际能力。

## 7. 稳健性要求

供应商层后续所有自动任务都应满足：

1. 幂等。
2. 不越层改写用户配置。
3. 只修供应商和供应商模型官方信息，不修用户运营决策。
4. 信息变更要能追溯到供应商官方网站或官方文档。
5. 能为 UI 提供“该模型已下线、已废弃、端点范围已调整、官方价格已调整”等解释信号。
