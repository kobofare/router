# 模型端点

本文档改为模型端点治理的总览入口。

之所以拆分，是因为原先把供应商层、渠道层、分组层放在同一篇文档里，容易混淆三个问题：

1. 哪一层拥有人工配置真值。
2. 哪一层只是官方信息来源、上游返回记录或静态展开结果。
3. 哪些自动任务可以写本层，哪些自动任务不能越层覆盖用户配置。

当前规范拆为三篇：

1. [供应商规范](./供应商规范.md)
2. [渠道规范](./渠道规范.md)
3. [分组规范](./分组规范.md)

## 核心总原则

1. 官方信息、上游返回记录、静态展开关系和人工配置真值必须分开理解。
2. 供应商层只负责供应商官方信息，不直接拥有渠道和分组的运营配置。
3. 渠道层是模型启用、端点启用的直接配置真值层。
4. 分组层是模型暴露范围和路由范围控制层，不拥有渠道端点开关。
5. refresh、migration、供应商信息更新都不能静默覆盖用户手工维护的模型和端点开关。
6. 自动禁用和人工禁用必须可区分，且 UI 需要能解释原因。
7. 管理端展示必须区分人工配置、上游返回记录、测试记录、解释信息，不能把它们混在一个字段里。
8. 已被分组接入的渠道，不能直接禁用，必须先处理分组引用或走显式联动处理。
9. 分组模型到渠道模型的静态展开关系，业务名称统一为 `group_model_channels`。

## 层间关系

### 供应商层 -> 渠道层

1. 供应商层输出官方模型信息、官方价格、官方端点范围。
2. 渠道层基于这些官方信息和上游返回记录生成可配置范围。
3. 供应商层不能直接替渠道层决定“启用哪个模型、启用哪个端点”。

### 渠道层 -> 分组层

1. 渠道层输出渠道人工配置和端点人工配置。
2. 分组层消费这些人工配置来刷新 `group_model_channels` 这种静态展开关系。
3. 分组层不能反向覆盖渠道层的模型与端点开关。

### 分组层 -> 用户接口层

1. `/v1/models` 只暴露当前用户所属分组可见的模型。
2. `/v1/models.supported_endpoints` 只暴露该分组下当前实际暴露的已启用端点。
3. 用户接口层不能直接看供应商官方信息或上游返回记录，必须经过渠道层与分组层处理后再输出。
