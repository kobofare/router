## Tokenizer 方案

### 待补充事项

1. 评估 tokenizer 依赖替换

- 当前项目使用 `github.com/pkoukk/tiktoken-go`，模型映射表较旧。
- 评估是否替换为更底层、只负责 encoding 的库，例如 `github.com/tiktoken-go/tokenizer`。
- 不再依赖第三方库内置的“模型名 -> encoding”映射完整性。

2. 自维护模型到 encoding 的映射

- 在 Router 内部维护一份清晰的 `model/prefix -> encoding` 规则表。
- 至少覆盖：
  - `gpt-5* -> o200k_base`
  - `o1* / o3* / o4* -> o200k_base`
  - `gpt-realtime* / gpt-audio* / gpt-image* -> o200k_base`
  - 老的 `gpt-4* / gpt-3.5*` 继续映射到现有 encoding

3. 评估是否引入官方 tokenizer 能力

- OpenAI 官方维护的是 Python 的 `tiktoken`，不是 Go 库。
- 评估是否需要通过 sidecar / 服务化方式接入官方 tokenizer，实现更准确的 token 估算。

4. 降级策略与日志治理

- 对未知模型不要继续回退到 `gpt-3.5-turbo`。
- 优先回退到同家族模型对应的 encoding。
- 对可预期的新模型家族，不打印错误级别日志，避免污染日志面板。

## 充值链路治理

### 当前状态

- 充值中心“前往充值”当前只是打开外部 `top_up_link`。
- 前端会附带 `username`、`user_id`、`transaction_id` 查询参数。
- Router 自身并不会在这一步创建充值单，也不会生成兑换码。
- 真正的入账动作仍然发生在用户拿到兑换码后，调用 `POST /api/v1/public/user/topup`。

### 待补充事项

1. 统一充值产品语义

- 明确区分“跳转外部充值页”和“Router 内部发放兑换码”两类模式。
- 避免页面文案、接口语义和实际业务行为不一致。

2. Router 自建充值单

- 设计内部充值单表，例如记录用户、订单号、金额、状态、来源渠道、回调信息。
- 提供创建充值单接口，而不是仅靠前端拼接 `transaction_id`。

3. 兑换码发放链路

- 如果需要“立即获取兑换码”，应明确由 Router 后台生成兑换码，还是由外部充值系统回传兑换码。
- 补齐充值成功后的兑换码发放、状态变更、幂等校验和失败补偿。

4. 充值记录统一展示

- 当前用户侧可直接展示充值日志。
- 后续如果引入充值单，应区分“充值订单记录”和“兑换入账记录”，避免把日志当业务主记录长期使用。
