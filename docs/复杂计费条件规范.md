# 复杂计费条件规范

本文档定义 Router 当前复杂计费能力中，`provider_model_price_components.condition` 字段的语义、写法和适用范围。

目标只有两个：

1. 让管理员在录入复杂计费时有统一规则，不靠口口相传。
2. 让后端匹配逻辑保持可预测，避免同一条价格规则在不同请求里行为不一致。

## 1. 适用范围

当前复杂计费已经接入以下链路：

- 图片模型计费
- 文本模型计费
- 音频模型计费
- 视频模型计费

当前尚未正式接入：

- realtime 更细粒度计费
- search / computer-use / tool 调用附加计费

## 2. 数据结构

复杂计费记录落在 `provider_model_price_components` 表，对应字段如下：

- `provider`: 供应商标识
- `model`: 模型名
- `component`: 计费项类型
- `condition`: 条件表达式
- `input_price`: 输入单价
- `output_price`: 输出单价
- `price_unit`: 计价单位
- `currency`: 币种
- `source`: 来源
- `source_url`: 官方来源链接

其中真正决定“什么时候命中这条规则”的核心字段只有两个：

- `component`
- `condition`

## 3. `component` 规范

当前支持的 `component` 取值如下：

- `text`
- `image_generation`
- `audio_input`
- `audio_output`
- `video_generation`
- `realtime_text`
- `realtime_audio`

当前系统已实际使用的有：

- `text`
- `image_generation`
- `audio_input`
- `audio_output`
- `video_generation`

其余值当前主要用于前向兼容和后续扩展，录入后不一定已经参与实际扣费。

## 4. `condition` 语法

### 4.1 基本语法

`condition` 使用简单的键值对表达式：

```text
key=value;key=value
```

规则：

- 多个条件用英文分号 `;` 分隔
- 单个条件使用 `=` 连接键和值
- 键和值都会按字符串比较
- 匹配时统一按小写比较
- 空条件表示“该 component 下的默认规则”

合法示例：

```text
endpoint=/v1/responses
quality=hd;size=1024x1024
size=1792x1024;quality=standard
```

不合法示例：

```text
endpoint:/v1/responses
quality
size=1024x1024,
```

### 4.2 匹配规则

- 所有键都必须命中，才算匹配成功
- 条件顺序不影响语义
- 未声明的键不会参与匹配
- 如果 `condition` 为空，则视为默认规则

例如：

```text
quality=hd;size=1024x1024
```

只有当：

- `quality=hd`
- `size=1024x1024`

同时成立时才会命中。

## 5. 各类模型当前支持的条件键

本节是当前代码已经支持并实际参与匹配的规范，不是规划。

### 5.1 文本模型

适用 `component`：

- `text`

当前支持的条件键：

- `endpoint`

可选值：

- `/v1/responses`
- `/v1/chat/completions`

说明：

- 文本模型的复杂计费匹配，基于最终上游端点，而不是下游原始请求路径。
- 如果下游调用 `/v1/chat/completions`，但 Router 最终桥接到上游 `/v1/responses`，则应按 `/v1/responses` 规则匹配。

示例：

```text
endpoint=/v1/responses
```

推荐用法：

- 给新版响应接口单独定价
- 给仍然走旧接口的渠道保留另一套价格

### 5.2 图片模型

适用 `component`：

- `image_generation`

当前支持的条件键：

- `quality`
- `size`

典型值：

- `quality=standard`
- `quality=hd`
- `size=1024x1024`
- `size=1024x1792`
- `size=1792x1024`

示例：

```text
quality=hd;size=1024x1024
```

说明：

- 当前图片复杂计费主要用于表达 OpenAI `dall-e-3` 这类按尺寸和质量分档的价格。
- 条件中键的顺序不重要，但建议统一写成 `quality=...;size=...`，便于人工阅读。

### 5.3 音频模型

适用 `component`：

- `audio_input`
- `audio_output`

当前支持的条件键：

- 暂无

说明：

- 当前音频复杂计费只按“输入型音频能力”与“输出型音频能力”区分，不解析 `condition`。
- `audio_output` 当前用于 TTS 等文本转音频场景。
- `audio_input` 当前用于转录、识别等音频转文本场景。

因此当前建议写法是：

```text
component=audio_input, condition=""
component=audio_output, condition=""
```

也就是不写条件，只靠 `component` 区分。

## 6. 默认规则与兜底规则

每个模型建议至少保留一条基础价格，放在 `provider_models` 主记录中。

如果有更细的复杂计费规则，再在 `provider_model_price_components` 中补充。

推荐优先级：

1. 先命中 `price_components`
2. 命不中时回退到模型主价格

因此复杂计费不是替代基础价格，而是覆盖基础价格。

## 7. 推荐写法

### 7.1 文本模型

基础价格：

- `provider_models.input_price`
- `provider_models.output_price`

复杂价格：

```text
component=text
condition=endpoint=/v1/responses
```

### 7.2 图片模型

基础价格：

- 录入一个默认图片价格，例如 `1024x1024 standard`

复杂价格：

```text
component=image_generation
condition=quality=standard;size=1024x1024

component=image_generation
condition=quality=hd;size=1024x1024
```

### 7.3 音频模型

基础价格：

- 保留模型基础单价

复杂价格：

```text
component=audio_input
condition=

component=audio_output
condition=
```

## 8. 当前不支持的条件键

以下条件键目前不要录入为正式规则，因为后端还不会识别：

- `mode`
- `voice`
- `format`
- `service_tier`
- `reasoning_effort`
- `tool`
- `search_context_size`
- `resolution`
- `duration`
- `seconds`
- `requests`

如果现在录入这些键，保存虽然可能成功，但运行时不会按预期命中。

## 9. 推荐约束

为了降低脏数据，建议管理员遵循以下约束：

- 所有键统一使用小写
- 所有枚举值统一使用小写
- 路径条件统一写完整路径，例如 `/v1/responses`
- 不要在一个模型下录入多条完全相同的 `component + condition`
- `source_url` 尽量填写具体定价页，而不是供应商首页

## 10. 后续扩展方向

后续如果继续扩展复杂计费，建议按以下顺序推进：

1. 文本模型支持更多条件键

- `service_tier`
- `reasoning_effort`
- `modality`

2. 音频模型支持更细分条件

- `mode=transcription`
- `mode=translation`
- `endpoint=/v1/audio/transcriptions`
- `endpoint=/v1/audio/speech`

3. 视频模型正式接入

- `component=video_generation`
- 支持时长、分辨率、任务模式等条件

4. realtime 计费接入

- `realtime_text`
- `realtime_audio`

## 11. 当前实现结论

截至当前版本，复杂计费的匹配能力可以概括为：

- 文本：按最终上游 `endpoint` 匹配
- 图片：按 `quality + size` 匹配
- 音频：按 `audio_input / audio_output` 类型匹配
- 视频：已支持 `video_generation` 的运行时匹配，并用于 `/v1/videos` 链路计费；更细的时长、分辨率、任务模式条件仍待扩展

这已经覆盖当前代码中最主要、最明确、最容易出错的复杂计费场景。
