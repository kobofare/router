# 里程碑

本文档用于沉淀 Router 的长期定位、阶段性发展方向和产品边界。

目标不是写成宣传稿，而是作为后续产品、研发、运营和文档演进的统一依据。

## 1. 当前判断

Router 未来不应只定位为“OpenAI 兼容转发器”或“多模型代理层”。

更准确的定位应该是：

**面向团队与运营场景的多模型接入、路由治理与商业化平台。**

对齐行业成熟多模型平台的部分：

- 一个 API 接入多个模型与供应商
- 路由、回退、协议兼容
- 统一模型目录与调用入口
- 基础可观测性与调用元数据

但 Router 不应停在“统一转发”这一层，而应该继续向上补足：

- 团队管理
- 业务运营
- 套餐与余额商品化
- 路由治理
- 成本治理
- 权限治理
- 私有化与本地化部署

## 2. 行业参考结论

行业成熟平台已经证明，“统一模型入口”这一层具备明确价值。  
同类产品普遍强调的核心方向包括：

- 模型目录与统一入口
- 供应商路由
- 自动路由
- 失败回退
- 请求级路由元数据
- 可观测性集成

Router 如果只做“支持更多模型”，产品价值会和既有统一模型入口高度重叠，且难以长期拉开差距。

因此 Router 应该把差异化明确建立在以下三件事上：

1. 运营治理
2. 路由治理
3. 私有化与本地化

## 3. 目标定位

### 3.1 一句话定位

Router 是一个面向团队与运营场景的多模型接入、路由治理与商业化平台。

### 3.2 北极星定义

> Router 不只是把请求转发到模型，  
> 而是把模型能力变成可售卖、可观测、可管控的业务资源。

### 3.3 对外可表达版本

可作为未来官网、README 或产品介绍的候选口径：

- 面向团队与运营场景的多模型接入与治理平台
- 面向团队、运营和私有化部署的模型资源治理平台
- 企业级多模型接入、路由治理与商品化平台

## 4. 差异化方向

### 4.1 运营治理

行业成熟平台通常更偏开发者入口层。  
Router 更适合成为团队和平台运营的控制层。

当前仓库已经具备这类能力基础：

- 用户、分组、套餐、余额、兑换码
- Topup、任务、业务流水
- Provider / 渠道 / 模型管理
- 管理后台与业务运营后台
- 用户分层、套餐可见范围、渠道健康、模型测试

因此 Router 可以自然扩展为：

- 用户分层运营平台
- 套餐与余额双计费平台
- 面向渠道和租户的模型开放平台
- 运营异常识别与处理闭环

### 4.2 路由治理

Router 不应只做“能路由”，而应做“可解释、可审计、可运营的路由系统”。

后续需要强调：

- 为什么命中这个 provider / channel / model
- 为什么发生 fallback
- 每次请求的选路链路
- 时延、成功率、成本、数据策略
- 用户 / 分组 / 套餐 / 组织维度的路由约束

换句话说：

- 统一模型入口更像 routing engine
- Router 更适合做 routing governance layer

### 4.3 私有化与本地化

这是 Router 最容易形成长期差异化的部分。

重点包括：

- 自部署
- 国内网络与支付环境适配
- 钱包登录 / 本地支付 / 本地供应商接入
- 面向代理商、团队主理人和二级运营的后台能力
- 企业内控、审计和权限体系

## 5. 目标用户

Router 不应假设只有“开发者”一种用户。

未来至少服务四类用户：

### 5.1 平台运营

关注：

- 哪些用户在高消费
- 哪些用户在沉默 / 流失
- 哪些套餐卖得出去
- 哪些模型与渠道在拖利润
- 哪些异常需要立刻处理

### 5.2 团队管理员 / 租户管理员

关注：

- 给成员开哪些模型
- 成本和预算怎么控
- 账号和权限怎么分
- 项目与部门怎么分账

### 5.3 渠道 / Provider 管理员

关注：

- 哪些渠道不稳定
- 哪些模型不兼容
- 哪些渠道该停用、重测或切流
- 哪些路由策略影响整体 SLA 和成本

### 5.4 接入开发者

关注：

- 协议统一
- 模型切换成本低
- 出错时容易定位
- 成本、速率和稳定性可预测

## 6. 产品边界

### 6.1 要做的

- 多模型统一接入
- 路由策略与回退治理
- 供应商 / 渠道 / 模型治理
- 团队、租户、套餐、余额、权限
- 可观测性、成本治理、运营动作闭环

### 6.2 暂时不作为主方向的

- 纯消费级聊天产品
- 资讯型模型目录站
- 仅以“支持更多协议 / 更多模型数量”为卖点
- 与主产品无关的大而全工作流编排

## 7. 三阶段路线图

## 7.1 阶段一：从转发器到控制台

目标：

让团队愿意把真实生产流量放进 Router。

核心建设：

1. 路由策略中心
   - Provider 优先级
   - Fallback 规则
   - 模型别名
   - 分组 / 套餐开放策略
   - 区域、数据保留、预算约束

2. 路由可观测性
   - 每次请求的真实命中链路
   - Fallback attempt 明细
   - 请求成功率、时延、成本、错误归因
   - 用户 / 模型 / 渠道 / 供应商视图

3. 成本治理
   - 每用户 / 每组 / 每套餐 / 每模型成本
   - 高成本模型限流与替代策略
   - 自动切换便宜 provider

阶段成果判断标准：

- 生产流量可持续跑
- 运营和研发能解释“为什么这么路由”
- 发生失败时能快速归因

## 7.2 阶段二：从控制台到运营系统

目标：

让运营、商务和团队管理员每天都会使用 Router。

核心建设：

1. 用户运营闭环
   - 用户画像
   - 用户分层
   - 高消费 / 高活跃 / 长尾 / 沉默识别
   - 批量加额 / 发码 / 分配套餐 / 导出名单

2. 商品化
   - 套餐模板
   - 模型能力包
   - 渠道化商品
   - 场景包，例如代码包、绘图包、推理包

3. 多租户与组织管理
   - 团队 / 项目 / 成员 / 角色
   - 项目级 token
   - 部门预算
   - 子账户分摊

阶段成果判断标准：

- 用户、套餐、余额和模型能力形成闭环
- 运营动作不需要离开 Router
- 商品化与路由策略能统一联动

## 7.3 阶段三：从运营系统到 AI 资源基础设施

目标：

把 Router 做成可扩展的 AI 接入基础设施，而不只是单体后台。

核心建设：

1. 渠道 / Provider 市场化基础能力
   - 渠道接入标准化
   - 模型能力发现
   - 健康度与质量评分
   - 价格与质量对比

2. 智能路由
   - 面向成本目标的自动路由
   - 面向 SLA 的自动路由
   - 面向套餐权益的自动路由
   - 面向组织策略的自动路由

3. 外部观测与审计集成
   - Langfuse
   - Helicone
   - 数据仓库
   - 企业审计系统

阶段成果判断标准：

- Router 对外成为“平台级中台”
- 外部系统能稳定集成
- 路由与治理逻辑可以标准化复用

## 8. 近 90 天建议

如果只看未来 2 到 3 个月，建议优先做以下三条：

详细执行计划见：`短期路线.md`。

### 8.1 Routing Governance

- 路由策略中心
- fallback / provider / region / data policy 策略
- 命中原因解释
- 路由失败链路审计

### 8.2 Ops Workflow

- 用户分层到用户列表的动作闭环
- 用户批量运营动作
- 套餐、余额、兑换码统一运营入口
- 异常用户 / 异常渠道 / 异常模型待办

### 8.3 Commercialization

- 套餐与模型权益绑定
- 组织 / 团队 / 子账户能力
- 成本、收入、毛利视图

## 9. 当前已具备的方向信号

从现有代码和后台结构来看，Router 已经具备以下发展信号：

- 有管理后台，不只是转发服务
- 有用户、余额、套餐和兑换码体系
- 有 Provider / 渠道 / 模型治理能力
- 有渠道健康、模型测试和任务体系
- 有业务运营入口，且已开始围绕运营视角优化页面

这说明 Router 不是“未来才做平台化”，而是已经走在平台化路上。

后续关键不在于“再做多少零散功能”，而在于：

**把这些功能收敛到清晰定位之下。**

## 10. 后续维护方式

本文档将持续优化，建议后续按以下方式迭代：

1. 每次产品方向调整时，优先更新本文档，再改具体页面和待办。
2. 新增模块前，先回答：
   - 它属于路由治理、运营治理、商品化还是基础设施？
   - 它是否强化了 Router 的差异化？
   - 它是否会把产品重心带偏？
3. 每次 roadmap 评审后，把“近 90 天建议”更新成最新版本。
4. 若未来出现更细的专题路线，可从本文档拆出子文档，但本文档保留为总纲。
